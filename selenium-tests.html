Need to quickly run selenium tests against multiple browsers?

Selenium is a great tool for running automated browser tests, butÂ it requires installation and maintenance, and is dependent on browser drivers installled in the host machine

Running Selenium-Server containerised solves this problem: images contain stable versions of the browser drivers that can be tested against in any machine, consistently.

This is a sample for the open source Sandbox tool

Run Selenium with a Sandbox workflow

There's an simple way still to dockerise Selenium. With Docker-based workflows, you can create a list of steps that automate both setting up Selenium-Server and running Selenium tests with the Selenium-Driver

Booting Selenium-Server using Sandbox

The workflows that Sandbox runs are just YAML files. Here's one that automates Booting up a Selenium Hub instance, plus two nodes connected to it - Firefox and Chrome.

Workflows define a sequence of steps. This one begins with a service step, which starts Selenium Hub, using an official Selenium Hub Docker image. 

steps:
  - service:
      name: Selenium Hub
      image: 'selenium/hub'

Normally all project source files are added to images created for steps. Here we request that source files are omitted for the Selenium Hub service. 

      source:
        exclude:
          - '*'

Here, the Selenium port is exposed "internally" to other workflow steps. It is exposed under the name hub, which allows other step containers to resolve that name using DNS (our app is going to use the url http://hub:4444/wd/hub). 

      ports:
        - name: 'hub'
          container: 4444
          internal: 4444
          external: 31044

We tell Sandbox to check that TCP port 4444 (the Selenium port) can be reached before considering the service ready. It will wait for the service to be ready before continuing. 

      readiness:
        tcp:
          port: 4444

The second and third steps run the browser nodes the Selenium Hub will use. They use official Docker images. By default, both will connect internally to http://hub:4444/wd/hub, which is the previous service image.

This step is a run step, with a parallel field set to true. This tells the workflow that the next step does not need to wait on this one to finish, so they will both run simultaneously. 

  - run :
      name: Selenium Chrome
      image: 'selenium/node-chrome' 
      parallel: true
      source:
        exclude:
          - '*'

The last step is a run step, as was the one before. It runs parallel to the previous step, and as it never completes, keeps the workflow running indefinitely. 

  - run :
      name: Selenium Firefox
      image: 'selenium/node-firefox'
      source:
        exclude:
          - '*'

Running Selenium tests against the running server

Now that the server is running, we can start testing using another workflow. This could also be done using a single workflow, but this way we can run the tests multiple times against the same Selenium-Server running instance.

The first step will install our testing application with npm, using an official Node Docker image. 

steps:
  - run:
      name: Install application
      image: 'node:8-alpine'
  - run:
      name: Run tests
      step: Install application
      source:
        exclude:
          - node_modules

We include only package.json from our source files, and aset cache to true. Doing this ensure the (potentially lengthy) installation step happens only once, and is redone only if dependencies have changed. 

      cache: true
      source:
        include:
          - package.json

Dependencies are installed by defining a script to be run on this image. 

      script: |-
        cd app
        npm install

The second step runs the tests. It uses as an image the one generated in the previous step, with the dependencies installed. As we expect dependencies to have been installed under node_modules, we exclude them from the included source files, preventing any overwriting. 

  - run:
      name: Run tests
      step: Install application
      source:
        exclude:
          - node_modules

The script field runs the tests for both browsers. 

      script: |-
        cd app
        SELENIUM_BROWSER=chrome \
        npm run start
        SELENIUM_BROWSER=firefox \
        npm run start

Run your tests immediately after checkout, no setup required!

The workflow above automates the entire process. And it can be run immediately after checkout if you commit both your workflow and Sandbox into your Git repo (Sandbox is tiny - under 200KB just so you can do this). No other software installation or setup is required! No databases, no caches, no Docker! And no long Wiki instructions required!

Try it yourself

We setup a sample repo at https://github.com/stackfoundation/sbox-selenium-tests so you can try how easy it can be yourself. Just clone and run the workflow using Sandbox, which is included in the repo! 

git clone https://github.com/stackfoundation/sbox-selenium-tests
cd sbox-selenium-tests
./sbox run start-server
[in a new terminal]
./sbox run selenium-tests
